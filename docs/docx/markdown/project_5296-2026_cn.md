CS5296 云计算，2026 春季

## 项目：
* 您将组成最多三人的小组。
* 您也可以选择个人独立完成项目。我们对个人项目有相同的期望，并遵循相同的评估规则（最终报告的页数限制除外）。
* 在小组或个人的基础上，您将选择并研究一个与云计算或边缘计算密切相关的问题/算法/技术/产品/公司（例如，移动应用、社交网站、新的编程语言、新的虚拟化技术、任务卸载、数字孪生、元宇宙、大语言模型 LLM、无人机 UAV 等）。

## 项目性质：
有三种项目选择：
* **研究类 (Research)**：您将研究一个技术问题，提出针对该问题的新解决方案，并对所提出的解决方案进行性能评估。解决方案可以是在现有解决方案基础上的改进，但必须有创新之处。研究类项目的最高分为 25 分。
* **技术类 (Technical)**：您将研究一个技术问题，并实现/评估现有的解决方案。技术类项目的最高分为 22 分。
* **业务类 (Business)**：您将研究一个业务、一家初创公司或某些技术的业务层面。业务类项目的最高分为 20 分。

## 交付物：
* 项目提案 (2 分)。
* 最终报告 (15 分)。
* 软件成果 (4 分)。
* 演示视频 (4 分)。

## 分组：
* 自行组队。只有在确定成员后，才在 Canvas 上填写分组信息。
* 前往 Canvas -> People -> 5296_groups 选项卡，那里已经创建了空的小组。将自己加入空小组或现有小组中。每个小组最多 3 名成员。

## 提案 (Proposal)：
* **目标**：帮助您尽早开始项目，而不是在最后一刻才动手。
* **内容**：
    * 您的项目主题
    * 您的项目性质
    * 主题简要摘要，包括其网站链接以及您将为该项目做什么
    * 组内每位成员的姓名和学号。对于个人项目，请在提案副标题写上“Individual project”。
* **格式**：
    * 最多 2 页 A4 纸大小，PDF 格式，单栏，单倍行距，12pt Times New Roman 字体。
    * 文件名应为：Group_ID_proposal.pdf，例如：Group_2_proposal.pdf。
* **提交**：
    * 截止日期：2026 年 2 月 6 日，23:59 (香港时间)。
    * 通过 Canvas -> Assignments -> Group projects -> Proposal 提交。仅允许提交 PDF 文件。不允许迟交。

## 最终报告 (Final Report)：
报告总结了您的项目发现。通常应包含以下部分（类似于学术论文）：
* **标题和作者**：在作者信息中包含您的小组 ID。
* **摘要**：项目的简短总结，少于 250 字。
* **引言**：介绍您的项目主题，展示背景，建立并阐明问题的重要性，介绍您的方法论（调查、在 EC2 上开发应用、Matlab 模拟等），并解释主要发现。呈现一个清晰、逻辑严密且有趣的故事。
* **报告主体**：报告主体可根据您的判断组织成一个或多个章节。您应详细介绍所研究的主题、您的新解决方案（如果有）、实验/实现的完成方式、性能评估/实验的观察结果，并解释这些观察结果。
* **结论**：用 3-4 句话总结您的项目。
* **参考文献**：包括您在报告中引用的任何论文/网站/来源。您可以按照 IEEE 格式组织参考文献。
* **成果附录 (Artifact appendix)**：描述如何重现您的项目步骤。详见下文“成果评估”部分。此部分不计入页数限制。

鼓励使用 LaTeX 编写报告，Word 编写的报告也可以接受。

* **格式**：A4 或 US Letter 大小（针对 LaTeX），PDF 格式，单栏，单倍行距，12pt Times New Roman 字体。
* **文件名**：GroupID_report.pdf。
* **页数限制**：小组项目 9 页，个人项目 8 页（包括参考文献）。

## 评分方案：
* (13 分) 内容：例如广度和深度、方案的新颖性、实现/实验/测量的难度和工作量等。
* (2 分) 写作：例如英语的使用、陈述的逻辑结构等。**严禁使用 ChatGPT 或其他大模型软件生成报告！**

## 提交：
* 截止日期：2026 年 4 月 24 日，23:59 (香港时间)。

## 软件成果 (Software Artifact)
成果评估旨在促进实验结果的可重复性。“成果”指在研究过程中创建或由实验本身产生的数字对象。例如，成果可以是软件系统/源代码、运行实验的脚本、输入数据集、实验收集的原始数据或用于分析结果的脚本。您需要在报告中使用专门的“成果附录”正式描述所有支持材料（代码、数据、模型、工作流、结果）。

**注意**：业务类项目没有软件成果（因此最高分较低）。

## 准备成果附录：
您需要准备成果附录，描述软件/硬件依赖关系、如何准备和运行实验以及预期的结果。您还需要提供至少一些构建工作流的脚本、运行工作流的所有输入以及一些验证结果的预期输出。

成果的提交和审查将在 GitHub 上进行。您应将成果（源代码、数据集等）托管在自己的 GitHub 仓库中。项目报告截止后，我们将 Fork 您的仓库到我们的仓库中，以确保代码真实反映报告中的描述。您的 GitHub 仓库应向公众开放。您还应定期使用 git 作为版本控制系统进行协作。

您可以制作自包含的 Docker 镜像、EC2 上的虚拟机镜像或 VirtualBox 虚拟机，以确保已安装所有依赖项。

* **格式**：与最终报告要求相同。
* **至少包含项目 GitHub 仓库的链接。**
* **页数限制**：成果附录没有页数限制，不计入报告最终页数。

## 评分方案：
* (0.5 分) 附录清晰完整，易于理解、安装和评估成果。
* (0.5 分) 使用便携式工作流框架（如容器或 VM 镜像、自动构建脚本等）使评估过程标准化、自动化和简化。
* (2 分) 成果被发现是一致、完整、可运行的，并包含适当的验证和确认证据。
* (1 分) 仓库应有不同成员在一段时间内的提交历史。最后一刻才提交的仓库将不会获得此分。

## 提交：
* 成果附录应与报告一起作为一个 PDF 文件通过 Canvas 提交。

## 演示视频 (Demo video)
您需要录制一段视频来演示您的项目。
1. **内容应包括**：
    * 项目的极简介绍。
    * 方案的总体解释。
    * 研究/技术类项目：运行成果（代码、测量等）的现场演示。
    * 业务类项目：调查的方法论。
    * 结果和 1-2 个重要发现。
2. 使用语音解释您的想法。
3. 可以通过添加字幕来编辑视频。
4. 视频分辨率不低于 640 * 480。
5. 演示时间不超过 10 分钟。
6. 演示使用的语言仅限英语。

## 评分方案
1. 内容 (1.5 分)：包含核心想法、实用的解决方案和清晰的运行指令。
2. 质量 (2.5 分)：突出并解释重点和创新思路。运行程序的指令应清晰易懂。遵守视频时长限制；有效使用字幕等编辑手段提高质量。

## 提交：
* 截止日期：2026 年 4 月 24 日，23:59 (香港时间)。
* 首先将演示视频上传到 YouTube 或 Bilibili，设为公开可见。
* 然后在 Canvas 上提交视频链接。上传到其他网站（如百度网盘）将直接计 0 分。
* 选出的优秀视频将分享给全班。
